# 客户端 UI/UX 深度走查建议报告

在完成整个客户端所有功能页面的走查后，整体架构非常清晰，基于 Tailwind + Wails 构建的桌面端体验流畅。以下是针对各个页面的 **UI/UX 与交互修改建议**，按照优先级划分。

---

## 🔴 高优建议 (影响核心体验与数据安全)

### 1. 发文页 (Publish.tsx) 草稿丢失问题
- **现状**：目前用户在左侧富文本编辑器中输入内容后，如果点击左侧导航栏切换到其他页面，**输入的内容会全部丢失**。
- **建议**：
  - 添加 `localStorage` 自动保存草稿机制。在 `onChange` 时将 `title`, `content`, `coverImage` 序列化保存。
  - 在组件 `useEffect` 初始化时尝试读取本地草稿，并提供“是否恢复上次未发布草稿”的提示。

### 2. 设置页 (Settings.tsx) 交互反馈缺失
- **现状**：“AI 发布辅助”区域的“保存配置”按钮（保存 Dify URL 和 API Key）在点击后，**没有任何成功或失败的反馈提示（Toast）**。
- **建议**：
  - 在 `handleSaveAIPublishConfig` 成功后添加 `toast.success('配置已保存')`。
  - 增加防抖机制，或者在保存时按钮显示 Loading 状态。
  - 统一保存逻辑：无头模式是自动保存，API Key是手动保存，建议在 UI 上做视觉区分（如提示“手动保存生效”）。

### 3. 冗余路由页面清理 (PublishTasks / PublishTaskDetail)
- **现状**：代码中存在 `PublishTasks.tsx` 和 `PublishTaskDetail.tsx` 作为独立页面路由，但侧边栏已经统一通过 `Tasks.tsx` (任务列表) 的 Tabs 切换来管理“搜索任务”和“发文任务”，并且任务详情已经使用了 Modal (`PublishTaskDetailModal`)。
- **建议**：
  - 移除冗余的独立页面路由，统一在 `Tasks` 面板中收敛。这样不仅减少了代码维护成本，还能避免用户通过意外路径进入导致状态丢失。

## 🟡 中优建议 (影响易用性与逻辑连贯性)

### 1. 搜索页 (Search.tsx) 任务创建后的跳转逻辑
- **现状**：点击“开始搜索任务”并成功创建后，只弹出一个 Toast 提示任务创建成功。用户想要查看进度，必须**手动点击左侧的“任务列表”**。
- **建议**：
  - 在 Toast 中加入 Action 按钮：“查看任务”。
  - 或者在创建成功后，自动执行 `navigate('/tasks')` 跳转，并激活“搜索任务” Tab。
  - “结果合并工具”需要手动输入用逗号分隔的任务 ID，门槛太高。建议改成一个抽屉/弹窗，提供多选列表（Checkbox），直接从已完成的搜索任务中勾选。

### 2. 账号授权页 (Auth.tsx) 列表扩展性问题
- **现状**：每个平台作为一个 Card 显示，账号列表直接在 Card 内部堆叠。当某个平台（如知乎或头条）授权了 **10+ 个账号**时，Card 的高度会无限撑长，破坏网格（Grid）布局的整齐度。
- **建议**：
  - 为账号列表区域添加最大高度与内部滚动条：`max-h-[250px] overflow-y-auto scrollbar-thin`。
  - 增加“暂无账号”的 Empty State 插画或占位符，现在如果没有账号只是一片空白。

### 3. 任务列表页 (Tasks.tsx) 分页与状态筛选
- **现状**：
  - 代码中写死了 `slice(0, 50)`，没有任何分页（Pagination）或“加载更多”控件。对于高频使用场景，用户无法查看 50 条之前的历史任务。
  - 原生的 `<select>` 下拉框在不同操作系统上的表现不一致，且不支持暗色模式深层定制。
- **建议**：
  - 引入 `shadcn/ui` 的 `Select` 组件替换原生 `<select>`。
  - 添加底部分页器或简单的“加载更多”按钮。

## 🟢 细节优化 (提升视觉品质)

1. **Dashboard 仪表盘空状态**：
   - 第一次打开应用加载时，数字会从 0 闪烁到实际数值。可以加个简单的数字滚动动画（如 `framer-motion` 或简单的 `useEffect` 递增）或加入 Skeleton 骨架屏。
   - 仪表板全是静态卡片，可以考虑将卡片做成可点击的（例如点击“失败”，直接跳转到 `/tasks` 并自动筛选 `status=failed`）。

2. **日志控制台 (Logs.tsx) 颜色高亮**：
   - 目前日志是纯文本显示。如果能用简单的正则表达式匹配 `[ERROR]` 显示为红色、`[INFO]` 显示为绿色、`[WARN]` 显示为黄色，将极大提升阅读体验。

3. **全局样式一致性**：
   - 在上一步走查中我已经通过脚本统一了 `shadow-sm`, `rounded-xl` 等 Tailwind 工具类，现在整个应用的卡片圆角和阴影是一致的。但要注意部分 `button` 仍有 `hover:scale-[1.02]` 的动画，尽量全局统一按钮的交互反馈（建议统一使用背景色加深的反馈，而非形变）。
